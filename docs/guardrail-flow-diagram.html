Guardrail flow

How requests move through the local guardrail proxy: NeMo validates user input, Presidio redacts input PII, Ollama generates a response, and Presidio checks output PII before the client receives the final text.

Function Of Each Layer

Layer 1: NeMo Input Guard (Input Policy Layer)

  • What it blocks: Prompt injection attempts, off-topic or unsafe user requests, and policy-violating input.
  • How it blocks: Runs a NeMo guardrail check using an LLM decision flow; if triggered, it returns a refusal message immediately and does not call upstream Ollama.

Layer 2: Presidio Input PII Guard (Input Privacy Layer)

  • What it blocks/modifies: Sensitive user data in prompts (for example email, phone, Taiwan national ID, Taiwan mobile, Taiwan student ID).
  • How it blocks/modifies: Uses Presidio entity detection with spaCy + regex recognizers; then either anonymizes detected values (default) or blocks depending on policy.

Layer 3: Ollama Generation Layer (Upstream LLM)

  • What it does: Generates the model response from the sanitized effective input.
  • How it is controlled: The proxy forwards a deterministic non-stream upstream call to ensure complete output can be checked before final return.

Layer 4: Presidio Output PII Guard (Output Privacy Layer)

  • What it blocks/modifies: Sensitive data in model outputs, including direct PII leaks or echoed user PII.
  • How it blocks/modifies: Runs Presidio detection on model output, then either replaces sensitive spans with redaction tokens or blocks the response text based on `pii_action`.

Layer 5: Proxy Response Layer

  • What it enforces: Final policy-consistent response to the client.
  • How it enforces: Returns input-block message when layer 1 intervenes, otherwise returns output after layer 4 processing.
